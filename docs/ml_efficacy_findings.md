# ML Efficacy Findings

_Read-only analysis of whether the XGBoost layer's predictions track reality
and correlate with trade P&L. Generated by `python -m scripts.ml_efficacy`
against the live (`:5434`) and paper (`:5433`) Timescale databases on
2026-06-02. No settings, trades, or schema were modified._

## TL;DR

- **The directional price model shows no edge.** Across both books, the
  sign of `predicted_return` was right ~**47–52%** of the time at a 7-day
  horizon — statistically indistinguishable from a coin flip (50%).
- **The probability model can't be scored at all.** `prediction_ledger` holds
  thousands of `predicted_prob` rows but **zero** backfilled
  `realized_outcome` values in both DBs — the outcome-marking step isn't
  running, so the calibration/Brier loop is open. The system can't even
  self-measure the directional model, let alone improve it.
- **P&L linkage is not yet measurable.** Only 5 (live) / 7 (paper) closed
  trades exist; you need ≥30 to say anything about confidence→P&L.
- **Verdict: don't rebuild — but the ML is not currently earning its rent,
  and the measurement loop is broken.** Fix the two gaps below, then re-run
  this report before making any keep/cut/rebuild decision.

## Interpretation

**Section A — directional probability model (unmeasurable).**
The agent *writes* predictions into `prediction_ledger` (predicted_prob is
populated), but nothing ever backfills `realized_outcome`. That backfill is
the first thing `scripts/nightly_calibrate.py` does — which means the nightly
calibration job is either not scheduled or not reaching the outcome-marking
step. Consequence: the probability model has never been scored against
reality, and `nightly_calibrate` has no outcomes to fit calibrators on, so the
whole calibrate→improve loop is inert.

**Section B — price/direction model (no edge).**
Over 522 evaluable predictions across both books, directional accuracy sits at
the coin-flip line (live 51.9%, paper 47.6%). The dollar errors (MAE
$31.91/$36.37) are absolute and should be read against each symbol's price
level, but the directional number is unit-free and unambiguous: **this signal
is not predicting next-week direction.** A 7-day horizon was assumed; if the
model's true target horizon differs, re-run with `--horizon-days N` — but a
mis-set horizon would not rescue a 47.6% reading.

**Section C — confidence → P&L (too early).**
5–7 closed trades is far too few to correlate model confidence with realised
P&L. This is the question that ultimately decides the ML's worth, and it
simply can't be answered yet. Re-run once ≥30 trades have closed.

## Recommended next steps (before any rebuild)

1. **Close the measurement loop.** Schedule the outcome backfill (run
   `scripts/nightly_calibrate.py` on the documented cron, or wire its
   outcome-marking into the agent) so `prediction_ledger.realized_outcome`
   gets populated. Without this, the directional model is permanently
   unmeasurable and the calibrators never fit.
2. **Re-run this report in ~2–4 weeks** once (a) outcomes are backfilled and
   (b) ≥30 trades have closed. Then Sections A and C produce real numbers and
   the keep/cut/rebuild decision can be made on evidence.
3. **Treat Section B's ~50% as a yellow flag, not a verdict.** It says the raw
   next-week direction signal has no edge today — but the strategies consume
   the model as a *POP overlay*, not a directional bet, so a separate question
   is whether the overlay helps or hurts trade selection (answerable via
   Section C once trade volume exists).

## How to reproduce

```bash
export HERMES_DSN="postgresql+psycopg://hermes:hermes@localhost:5434/hermes"  # or :5433 paper
python -m scripts.ml_efficacy                       # print report
python -m scripts.ml_efficacy --out docs/ml_efficacy_findings.md
python -m scripts.ml_efficacy --horizon-days 7 --min-rows 30
```

The script is read-only (SELECTs only) and exits 0 even on an empty DB
("insufficient data"), so it is safe to run against any environment.
